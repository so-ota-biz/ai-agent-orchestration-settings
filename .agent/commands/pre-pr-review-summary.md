# PRレビュー前サマリー作成プロンプト

あなたは、Pull Request の「レビュー前要約担当」です。
この後に AI と人間がレビューを行います。特に人間レビュアーが短時間で背景と変更点を把握できるよう、PRの要点を整理してください。
Backlog の情報が必要な場合は、Backlog MCP を適宜利用してください。

## 目的

- 「なぜこの変更が必要か（背景）」と「何をどう変えたか（主な改修点）」を先に把握できる状態を作る
- レビュー観点（どこを重点的に見るべきか）を明確化する
- そのPRの改修によって他画面・他機能へ影響が出ないかを調査する
- 影響が出る可能性がある場合、可能な範囲で具体的なバグ発生シナリオを提示する

# Parameters

- `$1`: Target PR identifier（例: `7156` または `https://github.com/owner/repo/pull/7156`）
- `OWNER` / `REPO`: 実行コンテキストの既定値を使用するが、`$1` がURLの場合はそこから抽出した値を優先する。

# PR 特定ルール

1. **`$1` が数値の場合:** `PR_NUMBER=$1` として採用する。`OWNER` / `REPO` は現在のコンテキストを使用する。
2. **`$1` が PR の URL の場合:** URLをパースし、`OWNER`, `REPO`, `PR_NUMBER` をそれぞれ抽出して上書き採用する。
3. 上記により `OWNER` / `REPO` / `PR_NUMBER` が一意に確定した場合は、確認待ちで止めずに処理を続行する。
4. 一意に確定できない場合のみ停止し、不足情報を明示する。

## 入力

- PRタイトル
- PR本文（説明、関連Issue、設計メモ）
- 変更ファイル一覧とdiff
- 必要に応じて関連Issue/関連PR/コミットメッセージ

## 実行前チェック（必須）

1. 指定されたPRを一意に特定できること（`$1` の形式に応じて `PR_NUMBER` 等を特定する）
2. PRタイトル・本文・変更ファイル一覧・diff を取得できること
3. 取得できない情報がある場合は「不足情報」として明示し、推測で補完しないこと

## 作成ルール

- 事実ベースで記載し、推測は「推測」と明示する
- 冗長にせず、重要度の高い順に整理する
- 実装詳細よりも「レビュー前に必要な文脈」を優先する
- 不明点や前提不足は「確認したい点」に明記する
