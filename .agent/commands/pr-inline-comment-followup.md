# Role

あなたは優秀なシニアフルスタックエンジニアです。

# Task

GitHub MCP を使用して、指定された PR に対するインラインコメント（修正指摘）への対応を完結させてください。
修正が必要な指摘はコードを修正し、修正が不要な指摘は不要と判断した理由を明確に報告してください。
Backlog の情報が必要な場合は、Backlog MCP を適宜利用してください。

# Parameters

- `$1`: Target PR identifier（例: `7156` または `https://github.com/owner/repo/pull/7156`）
- `OWNER` / `REPO`: 実行コンテキストの既定値を使用するが、`$1` がURLの場合はそこから抽出した値を優先する。

# PR 特定ルール

1. **`$1` が数値の場合:** `PR_NUMBER=$1` として採用する。`OWNER` / `REPO` は現在のコンテキストを使用する。
2. **`$1` が PR の URL の場合:** URLをパースし、`OWNER`, `REPO`, `PR_NUMBER` をそれぞれ抽出して上書き採用する。
3. 上記により `OWNER` / `REPO` / `PR_NUMBER` が一意に確定した場合は、確認待ちで止めずに処理を続行する。
4. 一意に確定できない場合のみ停止し、不足情報を明示する。

# Instructions

1. PR情報とコメントの取得

- GitHub MCP の `pull_request_read` を使って、対象PRの情報を取得する。
- 少なくとも以下を取得すること：
  - `method: "get"`（PR本文・メタ情報）
  - `method: "get_files"`（変更ファイルと行情報）
  - `method: "get_diff"`（差分本体）
  - `method: "get_review_comments"`（インラインコメント）
  - 必要に応じて `method: "get_reviews"`（レビュー状態の補足確認）

2. 対象コメントの選別

- インラインコメントのうち、修正指摘として対応が必要なものを抽出する。
- 明確に対応不要なもの（質問回答済み、前提誤認、すでに解消済み等）は、理由を記録したうえで対象外として扱う。
- 判定に必要な根拠が不足する場合のみ停止し、不足情報を明示する。

3. 修正方針の決定と実施

- 各指摘について、`対応方針（修正する / 修正しない）` を先に明確化する。
- 修正する場合は、対象ブランチ上で必要最小限のコード修正を行う。
- 指摘意図と差分がずれる修正は行わない。

4. 検証

- 変更した内容に直結する最小限の検証を実施する。
- 高コスト検証が必要な場合は実行前に確認する。
- 未実施検証がある場合は必ず明記する。

5. レビューコメントへの報告

- 各インラインコメントごとに、以下を明記して整理する。
  - 判定: `修正済み` / `修正不要`
  - 根拠: 事実ベースで簡潔に記載
  - `修正済み` の場合: 変更ファイルと要点、実施した検証
  - `修正不要` の場合: 不要と判断した理由
- PR上でのコメント返信は `修正不要` と判断したものに対してのみ行う。
- `修正不要` として返信する場合は、返信文の先頭でインラインコメント投稿者を必ずメンションする（例: CodeRabbitAI なら `@coderabbitai`、Codex なら `@codex`）。
- `修正済み` については、ローカル作業完了時点では該当コメントに返信しない。
- `修正不要` の返信が難しい場合は、最終報告に一覧でまとめる。

6. 完了報告

- 最終的に以下を明記する。
  - 対応したインラインコメント総数
  - `修正済み` 件数
  - `修正不要` 件数と理由要約
  - 実施した検証
  - 未実施検証と残リスク

# 完了通知

- すべての作業が終わった直後に、必ず以下のシェルコマンドを一度だけ実行してください。
  `sh ~/.agent/notification/notify.sh`
