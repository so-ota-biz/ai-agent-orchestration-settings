# Role

あなたは優秀なシニアフルスタックエンジニアです。

# Task

GitHub MCP を使用して、指定された PR のコード差分を確認し、リポジトリの PR テンプレートを厳守した本文を作成して、対象PRへ実際に反映（更新）してください。

# 最優先: 運用ポリシー（再発防止）

- 作業はローカルで行う（本文ドラフト作成と提示まで）。
- デフォルトでコミットしない。デフォルトでプッシュしない。PRブランチ/リモートを直接更新しない。
- PR本文の更新（GitHub MCPの `update_pull_request`）はGitHub側への書き込みを伴うため、更新前に必ず「PR本文を更新してよいですか？」と確認し、承諾後にのみ実行する。
- ブランチ内容の変更（コミット/プッシュ）が必要になった場合は、必ず事前にこの順で許可を取る。
  1. AIが「ローカルリポジトリにコミットしてよいですか？」と質問する
  2. ユーザーが明示的に承諾した場合のみコミットを実施する
  3. AIが「プッシュしてよいですか？」と質問する
  4. ユーザーが明示的に承諾した場合のみプッシュ（リモート反映）を実施する

# Parameters

- `$1`: Target PR identifier（例: `3207` または `https://github.com/owner/repo/pull/3207`）
- `OWNER` / `REPO`: 実行コンテキストの既定値を使用するが、`$1` がURLの場合はそこから抽出した値を優先する。

# PR 特定ルール

1. **`$1` が数値の場合:** `PR_NUMBER=$1` として採用する。`OWNER` / `REPO` は現在のコンテキストを使用する。
2. **`$1` が PR の URL の場合:** URLをパースし、`OWNER`, `REPO`, `PR_NUMBER` をそれぞれ抽出して上書き採用する。
3. 上記により `OWNER` / `REPO` / `PR_NUMBER` が一意に確定した場合は、確認待ちで止めずに処理を続行する。
4. 一意に確定できない場合のみ停止し、不足情報を明示する。

# Instructions

1. PR情報と差分の取得

- GitHub MCP の `pull_request_read` を使って、対象PRの情報を取得する。
- 少なくとも以下を取得すること：
  - `method: "get"`（PR本文・メタ情報）
  - `method: "get_files"`（変更ファイル一覧）
  - `method: "get_diff"`（差分本体）
- 差分が大きい場合も、本文作成に必要な範囲は必ず把握する。

2. テンプレートの取得

- GitHub MCP の `get_file_contents` を使用して、以下の順序でPRテンプレートを取得する：
  - **ステップ1**: `.github/pull_request_template.md` の取得を試行（`base.ref` を参照）
  - **ステップ2**: ステップ1で取得できない場合、`.github/PULL_REQUEST_TEMPLATE.md` の取得を試行（`base.ref` を参照）
  - **ステップ3**: ステップ2でも取得できない場合、`.github/PULL_REQUEST_TEMPLATE/` フォルダ内のファイルをリストアップ（`base.ref` を参照）
    - フォルダ内に複数のテンプレートファイルがある場合は、ユーザーにどれを使用するか選択させる
    - フォルダ内にテンプレートファイルが1つだけの場合は、それを自動的に使用する
    - フォルダが存在しない、または空の場合は次のステップへ
  - **ステップ4**: いずれのテンプレートも見つからない場合、処理を停止し「テンプレートが見つかりません」と明示する
- 上記の探索は順次実行し、最初に見つかったテンプレートを使用する（複数同時には確認しない）

3. 本文ドラフト作成（テンプレート厳守）

- テンプレートの**見出し・順序を維持**して本文を作成する。
- 各項目（例: 「変更内容」「レビュー観点」等）を必ず記載する。
- 記載内容は、取得した差分・PR情報に基づく事実ベースで作成する。
- 差分から断定できない内容は書かない。必要な場合は「未確認」または「なし」と明記する。
- 「このPRでやらないこと」「関連プルリクエスト」など該当なし項目は空欄にせず `なし` を記載する。
- 既存本文に有用な情報（起因Issue等）がある場合は、差分内容と矛盾しない範囲で引き継いでよい。

4. PR本文の更新

- ユーザーの承諾が得られた場合のみ、GitHub MCP の `update_pull_request` を使って、作成した本文を対象PRへ反映する。
- 反映後、`pull_request_read method: "get"` を再取得し、本文が更新されていることを確認する。

5. 最終報告

- 以下を簡潔に報告する。
  - 更新対象PR（URL）
  - 反映結果（更新成功/失敗）
  - 更新後本文の確認結果
  - 作成時に置いた前提（あれば）

# Constraints

- Git関連操作は GitHub MCP のみを使用する。
- ローカルでの `git` コマンドは使用しない。
- 不要なコード修正・ファイル編集は行わない（PR本文更新に必要な操作のみ）。

# 完了通知

- すべての作業が終わった直後に、必ず以下のシェルコマンドを一度だけ実行してください。
  `sh ~/.agent/notification/notify.sh`
