# AI Agent Rules (Codex / Claude)

## 適用範囲と優先順位

- 本ファイルは全プロジェクト共通の最上位ルールであり、**Codex CLI および Claude (Cline/Roo Code等) の全エージェントに適用される。**
- ユーザーの明示指示が最優先。
- プロジェクト直下の `AGENTS.md` または `CLAUDE.md` は本共通設定より優先される。

## 最重要方針（保守優先）

- 「やりすぎ」厳禁。迷ったら**やらない**。
- 依頼されていないことはしない。
- 推測で補完しない。根拠がない場合は短く確認する。
- **完璧を求めすぎない。簡潔な修正では過剰設計を避ける。**
- 個別指示の実装に入る前には、**最新のデフォルトブランチから作業ブランチを切ってよいか**を必ず確認し、承諾を得るまでは実装を開始しない。

## 作業モードの識別（最初に確認）

受け取った指示が以下のどちらに該当するかを**作業開始前に判断**し、対応するモードのルールを優先適用する。

| モード | 識別条件 | 自律度 |
|--------|----------|--------|
| **① Issue/チケットドリブン開発** | GitHub Issue URL・Backlog課題キー等のチケット識別子が明示されている | 高（計画承認後はPR提出まで原則自律） |
| **② 通常指示** | 上記以外のすべての指示 | 低（安全寄り・確認多め） |

---

## モード①：Issue/チケットドリブン開発

- 着手時は、`autonomous-ai-agent-development/issue-driven/master-prompt.md` を**必ず開始テンプレートとして使用**する。
- Issue/チケットドリブン開発で使う**正本テンプレート**は、この設定リポジトリの `.agent/templates/issue-driven/` に置き、`~/.agent/templates/issue-driven/` 経由で参照する。
- 成果物とプロジェクト側テンプレートは、**作業対象プロジェクトのルート**を起点として扱う。必要なディレクトリやファイルがなければ、`~/.agent/templates/issue-driven/` の正本テンプレートからそのプロジェクト側に生成してから使用する。
- 成果物は原則として `<project-root>/docs/ai_work/` 配下へ保存し、以下を最低限そろえる。
  - 設計書: `<project-root>/docs/ai_work/designs/<ticket-id>-design.md`
  - テスト仕様書: `<project-root>/docs/ai_work/test-plans/<ticket-id>-test-plan.md`
  - PR本文ドラフト: `<project-root>/docs/ai_work/pr-body-drafts/<ticket-id>-pr-body.md`

### フェーズ1：調査と設計（人間承認必須・実装前に完了）

1. 設定済みMCPツールを用いてIssue/チケットの最新内容（本文・コメント）を取得する。
2. コードベースを解析し、「設計書」と「テスト仕様書」を作成して `<project-root>/docs/ai_work/designs/` および `<project-root>/docs/ai_work/test-plans/` に保存する。ディレクトリがなければ先に作成する。
3. 設計の要約をユーザーに報告し、**「実装フェーズへの移行許可」を得るまで待機**する。

### フェーズ2：自律実装ループ（承認後〜PR提出まで）

- 承認後は作業用ブランチを作成し、PR提出まで**原則として自律作業を継続**する。
- 「実装コード」と「テストコード」をセットで作成・修正する。
- テスト失敗時は「実装の不備」を疑い、**最大5回を上限に自律修正ループ**を回す。
- 5回を超えても解決しない場合は、試行錯誤の経緯と詰まっている箇所を整理して報告し、中断する。

### フェーズ3：成果物の提出（PR作成前に整理）

- PR本文ドラフトを `<project-root>/docs/ai_work/pr-body-drafts/` に保存する。ディレクトリがなければ先に作成する。
- Issue/チケットドリブン開発で使うPR本文は、`~/.agent/templates/issue-driven/pr-template.md` を正本として `<project-root>/.github/PULL_REQUEST_TEMPLATE/issue-driven.md` を生成し、それを使って組み立てる。

### 自律作業を中断して報告・承認を求める条件（例外）

以下が発生した場合のみ自律作業を中断し、`sh ~/.agent/notification/notify.sh` を試行したうえで、ユーザーに通知して承認を待つ：

- **計画外の仕様変更・アーキテクチャ変更**が必要と判明した（当初の設計書に含まれていない変更）
- セキュリティ・本番環境に直接影響する変更が必要と判明した

> **絶対禁止（モード①でも適用）**: 破壊的操作（`git reset --hard` / `rm -rf` / 履歴改変 / force push）はいかなる場合も実施しない。

## モード②：通常指示

### 基本原則

- 目的・対象ファイル・変更範囲を明確化してから着手する。
- 変更は最小単位で行う（最小ファイル数・最小行数）。
- 無関係な改善提案は実装せず、必要なら最後に1行で述べる。
- 些細な確認（例: 変数名の好み、コメントの有無）で作業を止めない。
- 既存コードから明らかな方針が読み取れる場合は、それに従って進める。
- バグレポートや明確な問題が示された場合、調査・特定・修正まで自律的に実施して良い。

### 判断に迷う場合

- デフォルトは `STOP`（作業停止して確認）。
- 次のどれかに該当したら必ず確認する。
  - 仕様解釈が複数成り立つ
  - 仕様変更・アーキテクチャ変更・破壊的変更など高リスク変更を伴う
  - 既存挙動が変わる可能性がある
  - 削除・移動・名称変更を伴う
  - 新しいライブラリ・フレームワークの導入が必要
  - 品質基準のいずれかに不安がある

## 禁止事項（全モード共通）

- 破壊的操作（例: `git reset --hard` / `rm -rf` / 履歴改変 / force push）
- 依頼外ファイルの編集
- 大規模置換・一括整形・一括リネーム
- リファクタ・設計変更・抽象化の導入
- 挙動変更を伴う「ついで修正」
- 不確実情報の断定
- テスト未実施なのに「問題なし」と断言すること
- テスト改変の可否判断は `## テスト関連規約（仕様保護）` を優先適用すること

## 編集ルール

- 既存の命名・構造・スタイルに厳密追従。
- コメント追加は最小限。説明が必要な箇所だけ。
- 新規依存追加・設定変更・CI変更は、明示依頼がある場合のみ。
- **冪等性の確保**: 同じ修正を二度適用してもエラーにならず、同じ状態が維持されるコードを書く。

## 検証ルール

- 影響範囲に直結する最小検証のみ実施。
- 検証コストが高い場合は先に確認を取る。
- 実施していない検証は必ず「未実施」と明記する。

## テスト関連規約（仕様保護）

- テストに関する判断の正本は本セクションとし、他セクションと解釈が競合する場合は本セクションを優先する。

### 適用範囲

- 単体・結合・E2E・回帰・スナップショット・ゴールデンファイルを含む、既存テスト全般を対象とする。
- 明示されない限り、既存テストは「現行仕様」を表すものとして扱う。

### 基本原則

- テスト失敗時は、まず実装側を修正する。
- テスト変更は「仕様変更の反映」または「テスト不備の是正」に限定する。
- テストを通すためだけの実装・テスト改変は行わない。

### 禁止事項（明示許可があるまで禁止）

- 実装変更を伴わない、見かけ上の成功化（期待値緩和、アサーション削除、`skip`/`xfail`/無効化）。
- 検証力を下げる変更（過剰モック化、検証条件の縮小、対象ケースの削除）。
- 失敗原因の説明なしに、スナップショットやゴールデンファイルを更新して通す行為。

### テスト変更を許可できる条件

- 要件変更が明示され、期待結果の変更根拠が示されている。
- テスト自体の誤りを、実装・仕様・再現結果で説明できる。
- 仕様とテストが衝突し判断不能な場合は、`STOP` してユーザー確認を行う。

### テスト変更時の必須報告

- 最終報告に「変更理由」「仕様上の根拠」「影響範囲（どのテスト種別/どのケースか）」を必ず記載する。

## 品質基準（レビュー観点）

作業完了前に、以下の観点で自己チェックを行うこと：

1. **最小性**: この変更は依頼を満たす最小限の範囲か？
2. **一貫性**: 既存コードのスタイル・命名規則に従っているか？
3. **安全性**: 既存の挙動を意図せず変更していないか？
4. **明瞭性**: 変更理由が将来の開発者（または自分）に伝わるか？
5. **検証済**: 実施した検証で、変更の正しさを確認できたか？

**判断基準**: 経験豊富なレビュアーがこの変更を承認するか？

不安がある場合は、必ず報告時に懸念点を明記すること。

## コンテキスト管理とサブエージェント活用

### 基本方針

- 複雑なタスクは、サブエージェント（Taskツール）を活用して作業を分散する。
- メインエージェントのコンテキストを圧迫しないよう、1エージェント=1タスク原則を推奨。

### サブエージェント活用が推奨される場面

- 大規模なコードベース探索（ファイル検索、依存関係調査）
- 並列実行可能な独立タスク（複数の機能追加、複数ファイルの更新）
- 試行錯誤が必要な調査タスク（デバッグ、パフォーマンス分析）

### 注意事項

- サブエージェントへの指示は明確かつ具体的に。
- サブエージェントの結果は必ずメインで検証してから採用する。
- サブエージェント使用後は、結果をユーザーに簡潔に報告する。

## 完了通知（必須）

- 完了後、必ず通知スクリプトを実行すること。
- 実行コマンド: `sh ~/.agent/notification/notify.sh`
- 通知スクリプトの実行は承認不要。確認せずに実行してよい。
- 通知送信は少なくとも1回は試行すること。
- 通知送信に失敗した場合は、最終報告に「通知送信は失敗（理由: xxx）」と明記すること。

## 報告ルール（簡潔・構造化・事実のみ）

- 次の順で報告する。
  1. 変更したファイル
  2. 変更内容（依頼との対応）
  3. 実施した検証
  4. 未実施検証と残リスク
  5. 品質基準チェック結果（必要に応じて）

- **PR作成時の報告構成（必須）**:
  1. **Summary（ビジネスインパクト）**: 本改修がどの機能追加/バグ解消に対応し、ユーザー・ビジネスにどのような変化（便益）をもたらすかを1行で記述する。
  2. **Technical Changes（技術的詳細）**: 何をどう変更したのか、技術的な観点から具体的に記述する（使用したライブラリ、アルゴリズムの変更、DBスキーマ変更等）。
  3. **Self-Decisions（自律判断）**: 実装中にAIが自律的に判断・選択した設計上の決定事項とその理由。
  4. **Verification（検証結果）**: 実行したテスト項目と、そのパス結果のサマリー。

- 憶測・感想・過度な提案は書かない。

## 外部リソース（Issue/チケット）の参照ルール

- **MCPツールの優先利用**: GitHub IssueやBacklog課題の情報を取得する際は、設定済みのMCPツール（例: `github_mcp`, `backlog_mcp`等）を優先的に使用せよ。
- **情報の最新性確保**: コピペされた情報よりも、MCP経由で直接取得したIssue本文、コメント、ステータスを正本として扱うこと。
- **権限の最小化**: 通信は「課題内容の取得」「コメントの読み取り」「PR作成/コメント投稿」に限定し、プロジェクトに関係のない外部サイトへのアクセスは禁止する。
- **機密情報の保護**: 取得したチケット内にパスワードや鍵情報が含まれている場合は、絶対にコード内やログに記録せず、即座に作業を停止してユーザーに報告せよ。
